# 第一章：React 核心（最重要）

## 1.1 React 生命周期（三个版本演进）

### 概念理解

React 生命周期就是组件从"出生"到"成长"到"死亡"的全过程。不同版本只是流程变了，但核心思想没变：**挂载 → 更新 → 卸载**。

| 版本      | 阶段划分                     | 关键特点                   |
| --------- | ---------------------------- | -------------------------- |
| React 15  | Mount → Update → Unmount     | 同步递归更新，大树无法中断 |
| React 16  | Render → Pre-commit → Commit | 引入了 Fiber，可以中断     |
| React 17+ | 阶段合并，更安全             | concurrent mode 准备阶段   |

### 面试话术

```
React 生命周期经历了三个版本的演进，我分别说：

【React 15版本】
组件的生命周期分为三个阶段：挂载、更新、卸载。
挂载阶段依次执行：constructor → componentWillMount → render → componentDidMount
更新阶段分两种：props变化触发componentWillReceiveProps → shouldComponentUpdate → componentWillUpdate → render → componentDidUpdate
state变化触发：shouldComponentUpdate → componentWillUpdate → render → componentDidUpdate
卸载阶段：componentWillUnmount

【React 16/16.3版本】
引入了Fiber架构，将生命周期分为Render阶段和Commit阶段。
Render阶段可以被打断和恢复，包括：getDerivedStateFromProps、shouldComponentUpdate、render
Commit阶段不能被打断，包括：componentDidMount、componentDidUpdate、componentWillUnmount
还有个新的错误捕获：componentDidCatch

【React 17+版本】
主要是废弃了几个不安全的生命周期方法：
componentWillMount、componentWillReceiveProps、componentWillUpdate都被标记为UNSAFE_
因为异步渲染时，这些方法在render阶段可能被多次调用，导致逻辑混乱。

【我的理解】
其实核心就记住两点：
1. 以前是同步递归，一口气渲染完；现在Fiber架构可以"走走停停"，优先处理高优先级任务
2. getDerivedStateFromProps是个静态方法，无法访问this，更安全；它和componentDidUpdate配合基本能覆盖所有场景
```

### 追问预判

**Q: 为什么废弃 componentWillMount？**

```
因为React异步渲染时，componentWillMount可能在render阶段被多次调用。
比如在render里调用了setState，当前组件还没挂载完，又触发了新的render，
componentWillMount就会被调用多次，导致逻辑错误。

而且很多人喜欢在componentWillMount里请求数据，但这个时机并不靠谱，
网络慢的时候组件都卸载了数据才回来。所以官方推荐在componentDidMount里请求数据。
```

## 1.2 Hooks 原理

### 1.2.1 useState 原理

#### 概念理解

useState 就像一个**备忘录**，React 用链表记住每个状态值在哪里。setState 时，找到对应的 memoizedState 位置，更新值，然后触发重新渲染。

#### 面试话术

```
useState的实现原理，核心是链表和快照。

【初次渲染】
组件首次渲染时，React会创建一块内存（链表），按顺序存储每个状态的值。
每次调用useState，React从链表中取出一个位置，记录当前值和set函数。

【更新过程】
当调用setState时，React会标记当前组件需要更新，然后重新执行组件函数。
重新执行时，useState会从链表的下一个位置读取值（注意：这里用的是新值）。

【为什么setState是异步的】
setState不会立即更新状态，而是把更新请求放入一个队列。
React会在合适的时机批量处理这些更新，合并多次setState，减少渲染次数。
这也是为什么有时候连续调用两次setState({count: 1})和setState({count: 2)，结果只触发一次渲染。

【闭包陷阱】
useState容易踩的坑是闭包。初始值的count在render闭包里，如果你用setTimeout包裹setState，
过了2秒后拿到的count还是旧的。解决方法是使用setState的函数形式：setState(prev => prev + 1)

【🌰比喻】
可以把useState链表想象成电影的胶片，每一帧都是组件在某个时刻的"快照"。
React每次渲染就是"放映"某一帧。
```

#### 追问预判

**Q: 为什么 React.useState 的顺序很重要？**

```
因为React用链表按顺序记录状态。如果条件语句导致顺序不一致：
if (condition) {
  const [a, setA] = useState(1)
}
const [b, setB] = useState(2)

第一次渲染condition=true：链表[1, 2]
第二次渲染condition=false：链表[2]，但useState认为第二个位置的值是b，结果a的值被错误读取

所以Hooks不能在if、for、普通函数里调用。
```

### 1.2.2 useEffect 原理

#### 概念理解

useEffect = 副作用处理 = 组件渲染完成后"偷偷"做的事。React 用链表记录所有 effect，执行时机在 commit 阶段之后。

#### 面试话术

```
useEffect是处理副作用的钩子，比如数据请求、订阅、手动修改DOM等。

【执行时机】
useEffect在组件渲染和DOM更新完成后执行。
具体时机是commit阶段的"after mutation"阶段之后。
这意味着DOM已经更新了，但浏览器还没paint。

【依赖数组】
useEffect的第二个参数是依赖数组：
- 不传：每次渲染都执行
- 传[]：只在首次渲染执行（类似componentDidMount）
- 传[a, b]：a或b变化时执行

【cleanup函数】
useEffect可以返回一个cleanup函数，在下次effect执行前和组件卸载时调用。
这对于取消订阅、清除定时器非常有用。

【🌰例子】
const [count, setCount] = useState(0)

useEffect(() => {
  const timer = setInterval(() => {
    setCount(c => c + 1)
  }, 1000)

  return () => clearInterval(timer)  // cleanup
}, [])

这个effect只在首次渲染时执行，定时器会每秒更新count。
组件卸载时，cleanup函数会清除定时器，防止内存泄漏。

【常见坑】
1. 依赖数组写错，导致闭包陷阱或无限循环
2. 忘记cleanup，导致内存泄漏
3. 在effect里直接修改state导致无限循环
```

#### 追问预判

**Q: useEffect 和 useLayoutEffect 的区别？**

```
两者的区别在于执行时机：

useEffect：在commit阶段结束后，浏览器paint之前异步执行。如果有多个effect，按注册顺序执行。

useLayoutEffect：在commit阶段的DOM更新后，同步执行。用于需要立即读取/修改DOM的场景。

选择建议：
- 大部分情况用useEffect
- 如果effect里需要基于DOM尺寸做计算，或者需要立即修改DOM导致视觉变化，用useLayoutEffect
- useLayoutEffect可能会阻塞渲染，尽量避免在组件顶层使用
```

### 1.2.3 useCallback & useMemo

#### 概念理解

- useMemo：缓存**计算结果**，依赖变化才重新计算
- useCallback：缓存**函数引用**，依赖变化才返回新函数

本质上都是"记住"一个值，避免不必要的计算或重建。

#### 面试话术

```
useCallback和useMemo都是性能优化工具，用于避免不必要的计算或渲染。

【useMemo - 缓存计算结果】
const memoizedValue = useMemo(() => computeExpensiveValue(a, b), [a, b])

只有a或b变化时，才会重新执行computeExpensiveValue。
如果依赖没变，直接返回上次缓存的结果。

【useCallback - 缓存函数引用】
const memoizedCallback = useCallback(() => {
  doSomething(a, b)
}, [a, b])

只有a或b变化时，才会返回新的函数引用。
否则返回上次缓存的函数引用。

【为什么需要它们？】
React每次渲染，组件函数会重新执行，函数会被重新创建。
子组件props引用变化时，会认为props变了，从而重新渲染。

例如：
function Parent() {
  const [count, setCount] = useState(0)

  const handleClick = () => {  // 每次渲染都创建新函数
    console.log(count)
  }

  return <Child onClick={handleClick} />
}

Parent每次渲染，handleClick都是新引用，Child每次都会re-render。

用useCallback优化：
const handleClick = useCallback(() => {
  console.log(count)
}, [count])

只有count变化时，handleClick才是新引用。

【🌰比喻】
useMemo是"记住答案"，下次考试题目一样就不用重算了。
useCallback是"记住这支笔"，下次用同一支笔写字就不用换新的。
```

#### 追问预判

**Q: useMemo/useCallback 用多了有什么问题？**

```
useMemo/useCallback本身有开销：
1. 创建闭包
2. 存储返回值
3. 比较依赖是否变化

如果计算量很小，或者组件本来就会re-render，用useMemo/useCallback反而增加开销。

建议只在以下场景使用：
1. 传递给子组件的回调函数
2. expensive computation（计算量大的场景）
3. useMemo的返回值被多个地方引用
4. 配合React.memo使用

【最佳实践】
const memoizedValue = useMemo(() => {
  // 只在真正需要优化时才用
  return computeExpensiveValue(a, b)
}, [a, b])

不要为了"优化"而优化，先profiling定位瓶颈。
```

### 1.2.4 useRef 原理

#### 概念理解

useRef 就是创建一个"盒子"，这个盒子里的内容**永远不变**（引用稳定），但盒子里的值可以随意修改。

#### 面试话术

```
useRef返回一个ref对象，有current属性。

【两种用法】
1. 存储会在渲染间保持的值（不触发re-render）
2. 访问DOM元素

【用法1：不触发re-render的存储】
const timerRef = useRef(0)

const startTimer = () => {
  timerRef.current = setInterval(() => {
    setCount(c => c + 1)
  }, 1000)
}

const stopTimer = () => {
  clearInterval(timerRef.current)
}

timerRef.current的值变化不会触发re-render，因为ref不在React的数据流里。
这比useState更高效，适合存储定时器ID、DOM引用等。

【用法2：获取DOM引用】
const inputRef = useRef(null)

<input ref={inputRef} />

useEffect(() => {
  inputRef.current.focus()
}, [])

【useRef vs createRef】
createRef每次渲染都返回新ref，性能较差。
useRef只在首次渲染创建ref，后续渲染返回同一个ref。

【🌰比喻】
useRef就像一个带锁的保险箱。
你把东西放进去，箱子位置不变（引用稳定），但里面的东西可以随时换。
而且打开这个箱子不会让整个房子重新装修（不触发re-render）。
```

## 1.3 虚拟 DOM & Diff 算法

### 概念理解

虚拟 DOM 就是 JS 对象版本的 DOM 副本。Diff 就是对比新旧两个"副本"，找出最小变更，然后批量更新真实 DOM。

### 面试话术

```
【为什么需要虚拟DOM？】
直接操作真实DOM很慢，每次操作都可能触发重排重绘。
而且手动管理DOM状态很容易出错。

虚拟DOM的核心思想：
1. 用JS对象模拟DOM结构
2. 状态变化时，用新虚拟DOM与旧虚拟DOM对比（diff）
3. 找出最小变更
4. 批量更新真实DOM

这样减少了对真实DOM的直接操作次数。

【Diff算法核心策略】
React的Diff有三大策略：

1. tree diff：同层对比
跨层级的节点不会复用，直接销毁重建。
这意味着DOM节点不要频繁移动位置。

2. component diff：组件对比
同一类型的组件，递归对比子节点。
不同类型的组件，直接替换（连同子树一起销毁）。

3. element diff：元素对比
同一层级的多个子节点，用key来区分。
没有key时，只能按顺序对比，导致大量更新。
有key时，React会用key匹配，复用已有DOM节点。

【🌰例子】
原数组：[1,2,3]
新数组：[2,1,3]

没key：1→2，2→1，3不变（大量更新）
有key：1和2交换位置（移动）

【key的作用】
key必须是稳定且唯一的：
- ✅ 好：key={item.id}
- ❌ 坏：key={index}（index不稳定）
- ❌ 坏：key={Math.random()}（不唯一）

【为什么Diff快？】
因为是对象对比，不是真实DOM对比。
对象操作比DOM操作快几个数量级。
```

### 追问预判

**Q: Vue 的 Diff 和 React 有什么区别？**

```
Vue和React的Diff思想类似，都是同层对比，但实现细节不同：

React：从左到右对比，用index记录位置
Vue：双端对比，同时从新旧列表两端开始对比

Vue的策略更高效一些，但React 16+引入Fiber后，这个问题不那么突出了。
```

## 1.4 Fiber 架构

### 概念理解

Fiber 就是"光纤"，把渲染工作拆成小块，利用浏览器空闲时间执行。渲染可以被中断和恢复。

### 面试话术

```
【React 15的问题】
React 15使用同步递归渲染，组件树很大时，一次渲染可能耗时几百毫秒。
这期间主线程被阻塞，用户点击、输入都没反应，页面卡顿。

【Fiber的解决方案】
Fiber把渲染工作拆成小块（Fiber节点），每执行完一个小块就检查：
1. 有没有更高优先级的任务？（比如用户输入）
2. 浏览器是否需要控制权？（比如合成器滚动）

如果有，就中断当前渲染，把控制权还给浏览器，等浏览器空闲了再继续。

【双缓存技术】
React内部维护两颗Fiber树：
- current Fiber：屏幕上正在显示的树
- workInProgress Fiber：正在构建的新树

每次更新时，React在workInProgress上工作，完成后直接替换current。
这样用户看到的是完整的更新，没有半成品渲染。

【Fiber的工作流程】
1. beginWork：计算子节点，标记flag（ Placement、Update、Deletion等）
2. completeWork：向上冒泡，构建DOM，处理副作用
3. commitWork：统一更新真实DOM

【哪些任务有优先级？】
高优先级：用户输入、点击、滚动
中优先级：数据请求、动画
低优先级：预渲染、离屏内容
```

### 追问预判

**Q: React 18 的并发模式有什么新特性？**

```
React 18的并发特性：

1. automatic batching（自动批处理）
   之前只在事件处理器里批处理，React 18无论在哪里setState都会批处理
   setTimeout、Promise里setState也会合并

2. startTransition
   const [isPending, startTransition] = useTransition()
   startTransition(() => {
     setQuery(input)  // 低优先级
     setResults(data) // 高优先级
   })
   标记某些更新为"可中断的"，让紧急更新先执行

3. Suspense for data fetching
   结合服务端组件，数据获取更简单
```

## 1.5 合成事件机制

### 概念理解

合成事件就是 React 封装的"中间层"事件。你绑定的 onClick 不是原生的 click，而是 React 的事件代理。

### 面试话术

```
【为什么需要合成事件？】
1. 抹平浏览器差异，提供统一API
2. 所有事件都绑定到root元素，减少事件监听器数量
3. 方便实现事件冒泡、捕获、批量更新等

【工作原理】
React 17之前，所有事件绑定到root（document）
React 17+绑定到root（root元素）

事件绑定流程：
1. 事件触发时，document捕获到事件
2. 沿着DOM树向下"冒泡"，找到实际触发元素
3. 找到对应React组件，调用合成事件处理器
4. 合成事件包含原生事件对象的封装

【🌰例子】
function Button() {
  const handleClick = (e) => {
    console.log(e)  // 合成事件
    console.log(e.nativeEvent)  // 原生事件
    console.log(e.target)  // 真实DOM
    e.preventDefault()  // 阻止默认行为
    e.stopPropagation()  // 阻止冒泡
  }

  return <button onClick={handleClick}>点击</button>
}

【需要注意的点】
1. 事件池：React 16有事件池机制，事件对象会被复用。
   异步访问事件属性可能有问题（e.persist()可以解决）
2. 事件委托：React 17+不再挂载到document，减少与第三方库冲突

【与原生事件的区别】
onChange和原生change不一样：React的onChange在input、textarea、select的value变化时都会触发，包括blur事件
```

## 1.6 状态管理

### useContext vs Redux vs Zustand

### 概念理解

| 方案       | 适用场景                | 复杂度 | 性能                           |
| ---------- | ----------------------- | ------ | ------------------------------ |
| useContext | 主题、国际化等低频更新  | 低     | 中（任何变化都触发 re-render） |
| Redux      | 复杂状态、需要 devtools | 高     | 高（需要 selector 优化）       |
| Zustand    | 中等复杂度              | 中     | 高（细粒度更新）               |

### 面试话术

```
【useContext - 最简单的跨层传递】
Context适合"全局配置"类数据，比如主题、语言、用户信息。
但它的问题是：任何Context值变化，所有消费该Context的组件都会re-render。

优化方法：
1. 拆分Context：把变化频繁和变化少的分开
2. 使用useMemo包裹contextValue

【Redux - 复杂应用的首选】
Redux有三大原则：
1. 单一数据源
2. state只读，只能通过action修改
3. reducer必须是纯函数

核心流程：
action → reducer → newState → dispatch → UI更新

优点：
- 可预测、可追溯
- DevTools支持好（时间旅行调试）
- 中间件生态丰富（redux-thunk、redux-saga、redux-observable）

缺点：
- 模板代码多
- 需要额外优化（useSelector、React-Redux v7.1+）

【Zustand - 轻量级替代】
Zustand是Redux的轻量替代品，API更简洁：

const useStore = create((set) => ({
  count: 0,
  increase: () => set((state) => ({ count: state.count + 1 })),
  reset: () => set({ count: 0 })
}))

// 组件里直接用
function Counter() {
  const { count, increase } = useStore()
  return <button onClick={increase}>{count}</button>
}

优点：
- 无Provider包裹
- 无reducer模板
- 内置React 18的automatic batching
- 性能好（默认比较）

缺点：
- 没有Redux那么成熟（时间旅行等）
- 生态不如Redux

【我的选择建议】
- 小项目/简单状态：useContext + useReducer 或 Zustand
- 中型项目：Zustand
- 大型复杂项目：Redux + RTK Query
```

### 追问预判

**Q: Redux vs MobX？**

```
MobX是响应式状态管理，核心是observable、computed、action。
比Redux更"魔法"，上手简单但黑盒。

Redux更"显式"，所有状态变化都通过action追溯。
MobX适合快速开发，Redux适合大型团队协作、需要审计的场景。

个人更倾向Redux：显式优于隐式，状态变化可追溯在调试时很重要。
```

## 2.1 执行上下文 & 作用域链 & 闭包

### 概念理解

- **执行上下文**：代码执行时的"房间"，包含变量环境、作用域链、this
- **作用域链**：由近及远查找变量的"链条"
- **闭包**：函数记住并访问"出生时"外部变量的能力

### 面试话术

```
【执行上下文】
JavaScript代码在运行时，会创建一个执行上下文（Execution Context）。
就像给代码分配一个"房间"，房间里包含：
1. 变量环境（let、const、var声明的变量，函数声明）
2. 作用域链（Scope Chain）
3. this绑定

执行上下文分为三种：
- 全局执行上下文：浏览器里是window，Node里是global
- 函数执行上下文：每次调用函数创建
- Eval执行上下文：eval代码自己的上下文

【作用域链】
作用域链用于解析变量。
每个函数创建时，会记录它的[[Scope]]属性，指向创建时的作用域。
执行时，把所有外层作用域串联起来，形成作用域链。

查找变量时，沿着作用域链从近到远找：
1. 先在当前作用域找
2. 找不到，去外层作用域找
3. 重复，直到全局作用域
4. 找不到就报错

【🌰作用域链例子】
const a = 1
function outer() {
  const b = 2
  function inner() {
    const c = 3
    console.log(a, b, c)  // 都能找到
  }
  inner()
}

inner执行时作用域链：inner → outer → global

【闭包】
MDN定义：闭包是函数和该函数声明时词法环境的组合。

简单理解：函数能记住并访问"出生时"外部变量。

🌰经典例子：
function counter() {
  let count = 0
  return function() {
    return ++count
  }
}

const add = counter()
console.log(add())  // 1
console.log(add())  // 2
console.log(add())  // 3

counter执行完毕后，count应该被销毁。
但因为inner函数形成了闭包，count被"保护"起来了，外部无法访问，但inner还在用。

【闭包的应用场景】
1. 数据私有化
2. 函数工厂
3. 防抖、节流
4. 模块化

【闭包的问题】
1. 内存泄漏：闭包引用了大量外层变量，用完不释放
2. this指向：容易混淆

【⚠️ this指向问题】
function Counter() {
  this.count = 0
  setTimeout(function() {
    console.log(this.count)  // undefined，普通函数的this指向window
  }, 1000)
}

解决：
1. 箭头函数：箭头函数的this继承外层
2. bind/that保存
```

### 追问预判

**Q: 循环里的闭包问题？**

```
经典面试题：
for (var i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 100)
}
// 输出：5,5,5,5,5

因为var没有块级作用域，i是全局变量。
setTimeout回调执行时，循环已经结束，i=5。

解决：
1. let有块级作用域，每次循环创建新i
for (let i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 100)
}
// 输出：0,1,2,3,4

2. IIFE创建新作用域
for (var i = 0; i < 5; i++) {
  (function(j) {
    setTimeout(() => console.log(j), 100)
  })(i)
}

3. bind
for (var i = 0; i < 5; i++) {
  setTimeout(console.log.bind(null, i), 100)
}
```

## 2.2 原型链 & 继承

### 概念理解

原型链就是对象的"族谱"，每个对象都有个"爸爸"（prototype），"爸爸"还有自己的"爸爸"，直到 Object.prototype。

### 面试话术

```
【原型对象】
每个对象在创建时都有一个prototype属性，指向它的"原型对象"。
原型对象也有自己的prototype，形成链条。

访问obj.prop时：
1. 先在obj自身找
2. 找不到，去obj.__proto__找
3. 再去obj.__proto__.__proto__找
4. 直到Object.prototype
5. 找不到返回undefined

这个链条就是原型链。

【构造函数】
function Person(name) {
  this.name = name
}
Person.prototype.sayHi = function() {}

const p = new Person('张三')

new的过程：
1. 创建空对象 {}
2. 设置原型链：{}.__proto__ = Person.prototype
3. 构造函数this指向空对象
4. 执行构造函数
5. 返回对象

【继承的几种方式】
1. 原型链继承
Child.prototype = new Parent()
问题：共享引用类型，构造函数属性无法传参

2. 构造函数继承
function Child(...args) {
  Parent.apply(this, args)
}
问题：方法无法复用，每次创建都是新函数

3. 组合继承（最常用）
function Child(...args) {
  Parent.apply(this, args)
}
Child.prototype = new Parent()
Child.prototype.constructor = Child
问题：调用两次构造函数

4. 原型式继承（Object.create）
const child = Object.create(parent)
类似原型链继承，共享原型

5. 寄生组合继承（最佳）
function Child(...args) {
  Parent.apply(this, args)
}
Child.prototype = Object.create(Parent.prototype)
Child.prototype.constructor = Child

【ES6 class】
class Person {
  constructor(name) {
    this.name = name
  }

  sayHi() {
    console.log(`Hi, I'm ${this.name}`)
  }

  static create(name) {  // 静态方法
    return new Person(name)
  }
}

class Student extends Person {
  constructor(name, grade) {
    super(name)  // 调用父类构造函数
    this.grade = grade
  }

  sayHi() {
    super.sayHi()  // 调用父类方法
    console.log(`Grade: ${this.grade}`)
  }
}

class语法糖，底层还是原型链。
```

### 追问预判

**Q: instanceof 原理？**

```
instanceof检查构造函数的prototype是否在对象的原型链上。

function myInstanceof(obj, Constructor) {
  let proto = obj.__proto__
  while (proto !== null) {
    if (proto === Constructor.prototype) {
      return true
    }
    proto = proto.__proto__
  }
  return false
}

console.log([] instanceof Array)  // true
console.log([] instanceof Object)  // true（数组也是对象）
```

## 2.3 Event Loop（事件循环）

### 概念理解

JavaScript 是单线程的，靠 Event Loop 协调执行。同步代码在调用栈执行，异步代码等时机到了再执行。

### 面试话术

```
【为什么需要Event Loop】
JavaScript是单线程的，一次只能做一件事。
但有很多异步操作（网络请求、定时器、用户交互），不能干等着。
所以需要Event Loop来协调：主线程执行同步代码，异步代码"等通知"。

【调用栈（Call Stack）】
同步代码在这里执行，LIFO（后进先出）。
function a() { b() }
function b() { c() }
function c() { console.log(1) }
a()
执行顺序：a → b → c → console → c返回 → b返回 → a返回

【任务队列（Task Queue / Macrotask Queue）】
也叫宏任务队列，存放待执行的宏任务：
- setTimeout
- setInterval
- I/O操作
- UI渲染
- setImmediate（Node）

【微任务队列（Microtask Queue）】
- Promise.then/catch/finally
- MutationObserver
- queueMicrotask
- process.nextTick（Node）

【Event Loop流程】
1. 执行同步代码（调用栈）
2. 调用栈清空后，执行所有微任务
3. 微任务队列清空后，尝试DOM渲染
4. 取一个宏任务执行
5. 重复

【🌰经典题目】
console.log('1')

setTimeout(() => console.log('2'), 0)

Promise.resolve().then(() => console.log('3'))

console.log('4')

// 输出：1, 4, 3, 2

分析：
1. console.log('1') 同步执行
2. setTimeout是宏任务，进入任务队列
3. Promise.then是微任务，进入微任务队列
4. console.log('4') 同步执行
5. 调用栈空了，执行所有微任务：console.log('3')
6. 微任务队列空了，取一个宏任务：setTimeout里的console.log('2')

【async/await】
async函数返回值是Promise。
await会阻塞后面代码，但await后面的是同步执行的。

async function async1() {
  console.log('1')  // 同步
  await async2()
  console.log('2')  // await后面的，微任务
  await async3()
  console.log('4')  // await后面的，微任务
}

等价于：
function async1() {
  return new Promise(resolve => {
    console.log('1')
    resolve(async2())
  }).then(() => {
    console.log('2')
    return async3()
  }).then(() => {
    console.log('4')
  })
}
```

### 追问预判

**Q: setTimeout(fn, 0)不是立即执行？**

```
setTimeout(fn, 0)不是立即执行，而是"尽快执行"。
因为setTimeout是宏任务，需要等调用栈清空、微任务队列清空、DOM渲染后才会执行。

实际输出可能比0ms还慢，取决于当前队列情况。

应用场景：
1. 让出主线程，让浏览器先处理其他事
2. 在事件处理完后，延迟执行回调
3. 调整代码执行顺序
```

## 2.4 深拷贝/浅拷贝

### 概念理解

- **浅拷贝**：只复制第一层，嵌套对象还是共享引用
- **深拷贝**：完全独立，所有层级都复制

### 面试话术

```
【浅拷贝】
只复制对象的第一层属性。

方法：
1. Object.assign({}, obj)
2. {...obj}
3. arr.slice() / arr.concat()

问题：嵌套对象还是共享引用
const a = { b: { c: 1 } }
const d = { ...a }
d.b.c = 2
console.log(a.b.c)  // 2，被改了！

【深拷贝】
所有层级都独立复制。

方法1：JSON.stringify
const deepCopy = JSON.parse(JSON.stringify(obj))

问题：
- 函数、undefined、Symbol会丢失
- Date会变成字符串
- RegExp变成空对象
- NaN、Infinity变成null
- 循环引用会报错

方法2：手写递归
function deepClone(obj, map = new Map()) {
  if (obj === null || typeof obj !== 'object') return obj
  if (map.has(obj)) return map.get(obj)

  const clone = Array.isArray(obj) ? [] : {}
  map.set(obj, clone)

  for (const key in obj) {
    if (obj.hasOwnProperty(key)) {
      clone[key] = deepClone(obj[key], map)
    }
  }
  return clone
}

方法3：structuredClone（现代浏览器）
const clone = structuredClone(obj)
支持循环引用、Date、RegExp等

【🌰循环引用问题】
const a = { b: 1 }
a.self = a
JSON.stringify会报错，但手写递归用map记录可以处理。

【性能优化】
对于大对象，避免递归爆栈，可以用迭代方式：
- 先序列化再反序列化
- 使用lodash的deepClone
```

## 2.5 防抖/节流

### 概念理解

- **防抖（debounce）**：等用户停手才执行，最后一次
- **节流（throttle）**：控制执行频率，固定间隔

### 面试话术

```
【防抖 - debounce】
等用户"停手"一段时间后才执行。
如果用户一直动，就一直不执行，只执行最后一次。

应用场景：
- 搜索框输入：用户停止输入后才搜索
- 窗口调整：停止调整后才计算
- 表单验证：用户停止输入后才验证

实现：
function debounce(fn, delay) {
  let timer = null
  return function(...args) {
    clearTimeout(timer)
    timer = setTimeout(() => {
      fn.apply(this, args)
    }, delay)
  }
}

// 使用
const debouncedSearch = debounce(search, 300)

【节流 - throttle】
固定时间间隔执行一次。
无论用户动多快，保证每个周期最多执行一次。

应用场景：
- 滚动事件：滚动时做处理，但不能太频繁
- 按钮防连击：限制点击频率
- 拖拽：保证移动时不卡顿

实现（定时器版）：
function throttle(fn, delay) {
  let timer = null
  return function(...args) {
    if (timer) return
    timer = setTimeout(() => {
      fn.apply(this, args)
      timer = null
    }, delay)
  }
}

实现（时间戳版）：
function throttle(fn, delay) {
  let lastTime = 0
  return function(...args) {
    const now = Date.now()
    if (now - lastTime >= delay) {
      lastTime = now
      fn.apply(this, args)
    }
  }
}

【区别】
- 防抖：适合只需要最后一次结果的场景
- 节流：适合需要持续执行的场景

【🌰搜索框场景】
// 防抖：用户输入"vue"，期望只搜索一次"vue"
// 节流：滚动加载，用户快速滚动，希望每200ms检查一次是否到底

【优化：立即执行版】
有些场景希望第一次立即执行，最后一次等一段时间。
可以加immediate参数控制。
```

## 2.6 Promise & Async/Await

### 概念理解

Promise 是"承诺"，代表一个异步操作的最终结果。Async/Await 是 Promise 的"语法糖"，让异步代码看起来像同步。

### 面试话术

```
【Promise三种状态】
- pending（进行中）
- fulfilled（已成功）
- rejected（已失败）

状态一旦改变，就不可逆。

【Promise基本用法】
const promise = new Promise((resolve, reject) => {
  // 异步操作
  if (success) {
    resolve(result)
  } else {
    reject(error)
  }
})

promise
  .then(result => console.log(result))
  .catch(error => console.error(error))
  .finally(() => console.log('完成'))

【Promise链式调用】
then、catch、finally都返回新Promise。
返回值会传递给下一个then。

return的值：
- 非Promise值：直接传递给下一个then
- Promise：等待Promise resolve/reject后继续

new Promise((resolve, reject) => {
  resolve(1)
})
  .then(res => {
    console.log(res)  // 1
    return res + 1
  })
  .then(res => {
    console.log(res)  // 2
    return Promise.reject('error')
  })
  .then(res => {
    console.log('不会执行')
  })
  .catch(err => {
    console.log(err)  // 'error'
    return 'recovered'
  })
  .then(res => {
    console.log(res)  // 'recovered'
  })

【Promise静态方法】
1. Promise.all([p1, p2, p3])
   全部成功才成功，有一个失败就失败
   返回按顺序的结果数组

2. Promise.allSettled([p1, p2, p3])
   全部执行完毕才返回，不管成功失败
   返回结果数组，每个有status: 'fulfilled'/'rejected'

3. Promise.race([p1, p2, p3])
   返回最快执行完的（成功或失败）

4. Promise.any([p1, p2, p3])
   返回最快成功的，都失败才失败

【async/await】
async函数返回值是Promise。
await暂停async函数执行，等待Promise resolve/reject。

async function fetchData() {
  try {
    const res = await fetch('/api/data')
    const data = await res.json()
    return data
  } catch (error) {
    console.error('请求失败', error)
  }
}

【错误处理】
async/await的错误用try/catch处理。
但要注意：
- try/catch只能捕获同步错误和await的Promise reject
- fetch错误（网络断开）需要判断res.ok

【常见面试题：Promise执行顺序】
console.log('1')

setTimeout(() => console.log('2'), 0)

new Promise((resolve, reject) => {
  console.log('3')
  resolve()
}).then(() => console.log('4'))

console.log('5')

// 输出：1, 3, 5, 4, 2
// 解释：Promise构造函数是同步的，只有then是异步
```

### 追问预判

**Q: Promise.then 是微任务还是宏任务？**

```
Promise.then是微任务。

Node.js事件循环阶段：
   ┌───────────────────────────┐
┌─>│         timers            │  setTimeout, setInterval
│  └───────────────────────────┘
│  ┌───────────────────────────┐
│  │   pending callbacks       │  I/O callbacks
│  └───────────────────────────┘
│  ┌───────────────────────────┐
│  │     idle, prepare         │  内部使用
│  └───────────────────────────┘
│  ┌───────────────────────────┐
│  │         poll              │  获取新的I/O事件
│  └───────────────────────────┘
│  ┌───────────────────────────┐
│  │         check             │  setImmediate
│  └───────────────────────────┘
│  ┌───────────────────────────┐
│  │    close callbacks        │  socket.on('close')
└──┘

微任务：
- 每个阶段结束后，**所有**微任务都会执行
- process.nextTick优先级高于Promise.then

执行顺序：
timers → check后执行微任务 → poll → check → close → timers...

所以在Node里，setImmediate和Promise.then的顺序不确定，取决于poll阶段情况。
在浏览器里，Promise.then先于setTimeout(0)执行。
```

## 3.1 interface vs type 区别

### 概念理解

interface 是"合同"，定义对象长什么样；type 是"别名"，给复杂类型起个简单名字。

### 面试话术

```
【共同点】
都能定义对象类型：

interface Person {
  name: string
  age: number
}

type Person = {
  name: string
  age: number
}

都能被implements：
class Student implements Person {}

都能被extends：
interface Teacher extends Person {}

【区别】

1. 声明合并
interface可以重复声明，合并：
interface Window {
  title: string
}
interface Window {
  ts: TypeScriptAPI
}
// 相当于 Window { title: string, ts: TypeScriptAPI }

type不行，重复声明会报错：
type Window = { title: string }  // Error

2. 计算属性
type可以用映射类型：
type Keys = 'name' | 'age'
type Person = {
  [K in Keys]: any
}
// { name: any, age: any }

interface不行。

3. 联合类型
type可以定义联合类型：
type ID = string | number
type Result = Success | Error

interface不行直接定义联合。

4. 元组
tuple用type更清晰：
type Tuple = [string, number, boolean]

5. 使用场景
建议：
- 定义"对象/类"的形状：interface
- 定义联合类型、交叉类型、元组、映射类型：type
- 需要声明合并（如给全局对象扩展）：interface

【🌰我的项目实践】
// interface用于定义API响应
interface ApiResponse<T> {
  code: number
  data: T
  message: string
}

// type用于定义业务联合类型
type FormType = 'input' | 'select' | 'date' | 'cascader'
type UserRole = 'admin' | 'teacher' | 'student'
```

## 3.2 泛型

### 概念理解

泛型就是"类型变量"，让函数/接口/类型在调用时才知道具体类型。

### 面试话术

```
【为什么需要泛型？】
不用泛型时，类型写死，不灵活：
function identity(arg: number): number {
  return arg
}
只能传number。

用any可以传任何类型，但丢失类型信息：
function identity(arg: any): any {
  return arg
}
返回类型是any，调用identity(1).length不报错（因为any不报错）。

泛型登场：
function identity<T>(arg: T): T {
  return arg
}
调用时可以指定类型：
const result = identity<string>('hello')
// result是string类型

或者让TypeScript自动推断：
const result = identity('hello')  // result是string

【泛型函数】
function swap<T, U>(tuple: [T, U]): [U, T] {
  return [tuple[1], tuple[0]]
}

【泛型接口】
interface KeyValuePair<K, V> {
  key: K
  value: V
}

【泛型约束】

  return arg.length
}

getLength(123)      // Error，数字没有length

【extends关键字在泛型中】
T extends U 表示T是U的子类型。

  return obj[key]
}

  message: string

  // ...
}
```

## 3.4 类型守卫

### 概念理解

类型守卫就是"类型检查员"，帮 TypeScript 缩小联合类型的范围。

### 面试话术

```
【什么是类型守卫】
当有联合类型时，TypeScript不知道运行时是哪个具体类型。
类型守卫让TypeScript在if块里自动收窄类型。

【常见类型守卫】

1. typeof
只适用于基本类型：
function process(value: string | number) {
  if (typeof value === 'string') {
    // value在这里是string
  }
}

2. instanceof
适用于类：
class Dog { bark() {} }
class Cat { meow() {} }

function makeSound(animal: Dog | Cat) {
  if (animal instanceof Dog) {
    animal.bark()
  } else {
    animal.meow()
  }
}

3. in操作符
适用于对象属性：
type Fish = { swim: () => void }
type Bird = { fly: () => void }

function move(animal: Fish | Bird) {
  if ('swim' in animal) {
    animal.swim()
  } else {
    animal.fly()
  }
}

4. 自定义类型守卫
function isString(value: unknown): value is string {
  return typeof value === 'string'
}

使用：
function process(value: unknown) {
  if (isString(value)) {
    // value是string
    console.log(value.toUpperCase())
  }
}

5. 可辨识联合（Discriminated Unions）
给每个类型加个共同的"标签"属性：
interface Circle {
  kind: 'circle'
  radius: number
}
interface Square {
  kind: 'square'
  side: number
}
type Shape = Circle | Square

function area(shape: Shape) {
  switch (shape.kind) {
    case 'circle':
      return Math.PI * shape.radius ** 2
    case 'square':
      return shape.side ** 2
  }
}

【🌰项目实践】
// 统一错误处理
type Success<T> = { success: true; data: T }
type Fail = { success: false; error: string }
type Result<T> = Success<T> | Fail

function handleResult<T>(result: Result<T>) {
  if (result.success) {
    // TypeScript知道这里是Success<T>
    console.log(result.data)
  } else {
    // TypeScript知道这里是Fail
    console.error(result.error)
  }
}
```

# 第四章：Webpack/工程化（简历亮点）

## 4.1 Webpack 构建流程

### 概念理解

Webpack 就是"打包工厂"，把各种资源（JS、CSS、图片）加工成浏览器能用的静态文件。流程：初始化 → 编译 → 输出。

### 面试话术

```
【Webpack核心流程】

1. 初始化阶段（Initialization）
- 读取配置：webpack.config.js
- 创建Compiler对象（一个compiler对应一个webpack配置）
- 初始化插件（plugins在compiler上注册）

2. 编译阶段（Compilation）
- 从entry开始，递归解析所有模块
- 每个模块调用loader处理
- 生成模块依赖图（Module Dependency Graph）

3. 输出阶段（Emit）
- 把模块打包成chunks
- 生成最终文件
- 写入文件系统

【🌰详细流程】
1. 调用webpack(options)，生成Compiler
2. 调用compiler.run()，开始构建
3. 从entry配置开始：
   entry -> [loader解析] -> 模块对象
4. 递归解析模块的依赖：
   模块A依赖B -> B依赖C -> 解析C -> B -> A
5. 调用plugins的钩子（tap）：
   emit钩子：生成文件前
   afterEmit钩子：生成文件后
6. compiler.emitAssets()：写入文件
7. compiler.close()：完成

【关键概念】
- Compiler：整个构建过程的控制中心
- Compilation：单次编译过程，包含当前模块、依赖、生成资源
- Module：每个源文件解析后的模块对象
- Chunk：代码块，入口文件和动态import生成
- Bundle：最终输出的文件
```

### 追问预判

**Q: Webpack 为什么选择这些阶段暴露钩子？**

```
Webpack基于Tapable实现事件流，和Node的EventEmitter类似但更强大。

暴露钩子是为了让插件在特定时机介入：
- beforeRun：开始构建前，清理上次产物
- run：开始读取records
- compile：创建Compilation之前
- compilation：创建Compilation
- emit：生成资源到output目录之前
- afterEmit：生成资源到output目录之后
- done：构建完成

Plugin就是"订阅"这些钩子，在合适时机执行。
```

## 4.2 Loader vs Plugin 区别

### 概念理解

- **Loader**：翻译员，把非 JS 文件翻译成 Webpack 认识的东西
- **Plugin**：记者，在 Webpack 生命周期里做各种事情

### 面试话术

```
【Loader - 模块转换器】
Webpack原生只能处理JS。
Loader让Webpack能处理其他类型的文件。

loader特点：
1. 链式调用，一个loader处理完可以传给下一个
2. 转换文件内容
3. 从右到左执行，从下到上

🌰配置：
module: {
  rules: [
    {
      test: /\.css$/,
      use: ['style-loader', 'css-loader', 'postcss-loader']

  ]
}
// 执行顺序：postcss-loader -> css-loader -> style-loader
// postcss-loader先把CSS处理一下 -> css-loader处理@import -> style-loader插入DOM

常见loader：
- css-loader：处理@import等CSS语法
- style-loader：把CSS注入到DOM
- babel-loader：ES6+转ES5
- ts-loader：TypeScript编译
- file-loader/url-loader：处理图片/字体等
- vue-loader：处理Vue单文件组件

【Plugin - 扩展器】
Plugin可以介入Webpack整个构建过程。

Plugin特点：
1. 基于Tapable事件流
2. 在特定生命周期执行
3. 可以访问/修改编译结果

常见plugin：
- HtmlWebpackPlugin：生成HTML文件
- MiniCssExtractPlugin：抽离CSS到单独文件
- DefinePlugin：定义全局常量
- CopyWebpackPlugin：复制静态资源
- UglifyjsPlugin：压缩代码
- CleanWebpackPlugin：清理output目录

【🌰手写Plugin】
class MyPlugin {
  apply(compiler) {
    // emit钩子：生成文件时
    compiler.hooks.emit.tap('MyPlugin', (compilation) => {
      for (const filename in compilation.assets) {
        console.log(`生成文件: ${filename}`)
      }
    })
  }
}

module.exports = MyPlugin
```

## 4.5 Tree Shaking 原理

### 概念理解

Tree Shaking 就是"摇树"，把没用的代码"摇掉"。通过静态分析，删除没用的 export。

### 面试话术

```
【原理】
Webpack 5基于ES Module的静态结构进行Tree Shaking。
ESM在编译时就能确定导入导出，而CommonJS是动态的。

【🌰例子】
// util.js
export function add(a, b) { return a + b }
export function minus(a, b) { return a - b }

// main.js
import { add } from './util'
console.log(add(1, 2))
// minus函数没被引用，会被删除

【生效条件】
1. 使用ES Module（import/export）
2. mode: production 自动开启
3. 不能有副作用（sideEffects）

sideEffects配置：
// package.json
{
  "sideEffects": ["*.css", "*.scss"]  // 这些文件有副作用，不要删除
}

// webpack.config.js
{
  module: {
    rules: [
      {
        test: /\.js$/,
        use: {
          loader: 'babel-loader',
          options: {
            presets: [
              ['@babel/preset-env', { modules: false }]  // 保留ES模块
            ]
          }
        }
      }
    ]
  }
}

【⚠️ 注意事项】
1. babel要配置 presets: [['@babel/preset-env', { modules: false }]]
   否则Babel会把ESM转CommonJS，Tree Shaking失效

2. 函数式代码如果有副作用，不能Tree Shaking
   if (process.env.NODE_ENV === 'production') {
     console.log('hello')  // 有副作用，不能删除
   }

3. webpack5的sideEffects比4更智能
```

# 第五章：浏览器原理

## 5.1 渲染过程（重排/重绘）

### 概念理解

浏览器渲染页面就是：解析 HTML → 计算样式 → 布局 → 绘制 → 合成。

### 面试话术

```
【浏览器渲染流程】

1. HTML解析 → DOM树
2. CSS解析 → CSSOM树
3. DOM + CSSOM → Render Tree
4. Layout（布局）：计算每个节点几何信息
5. Paint（绘制）：绘制每个节点到图层
6. Composite（合成）：把图层合成为最终图像

【关键概念】
- DOM Tree：解析HTML
- CSSOM Tree：解析CSS
- Render Tree：DOM+CSSOM结合
- Layout：计算位置和大小
- Paint：绘制像素
- Composite：合成层

【回流（Reflow/Layout）】
几何属性变化时触发，如：
- 元素大小、位置变化
- 添加/删除可见元素
- 窗口大小变化
- 字体大小变化

【重绘（Repaint）】
外观变化但不影响布局，如：
- 颜色变化
- 背景变化
- visibility变化

【🌰回流必定重绘，重绘不一定回流】

【优化策略】
1. 批量修改DOM
   - 一次性修改，用innerHTML
   - 或先display:none，改完再show
   - 记录修改，最后一起apply

2. 使用transform/opacity
   transform和opacity不触发回流重绘，只触发合成
   // 差：element.style.top = top + 'px'  // 触发回流
   // 好：element.style.transform = `translateY(${top}px)`  // 只合成

3. 避免频繁读写
   读写分离，不要交替：
   // 差：读取offsetHeight后立即设置width
   // 好：先读所有，再写所有

4. 使用will-change
   提前告诉浏览器元素会变化：
   .card { will-change: transform }

5. 使用CSS containment
   .container { contain: layout paint }
```

## 5.3 缓存机制

### 概念理解

浏览器缓存就是"本地仓库"，存着常用的资源，下次请求直接用，不用再下载。

### 面试话术

```
【缓存位置】
1. Service Worker：运行在浏览器后台，可以编程控制缓存
2. Memory Cache：内存缓存，关闭浏览器就没了
3. Disk Cache：磁盘缓存，持久存储
4. Push Cache：HTTP/2的推送缓存

【缓存优先级】
1. 先检查Service Worker（如果有）
2. 再检查Memory Cache
3. 再检查Disk Cache
4. 发送网络请求

【强缓存（Cache-Control）】
不发送请求，直接使用本地缓存。

Cache-Control: max-age=3600
- max-age：缓存有效秒数
- no-cache：不用本地缓存，总是协商
- no-store：彻底禁用缓存
- private/public：private只能浏览器缓存，public可被代理缓存

Expires: Wed, 21 Oct 2025 07:28:00 GMT
- 绝对时间，已被Cache-Control取代

【协商缓存（Last-Modified / ETag）】
发送请求，服务端判断是否使用缓存。

Last-Modified / If-Modified-Since：
- 服务端返回Last-Modified
- 下次请求带上If-Modified-Since
- 服务端对比时间，相同返回304

ETag / If-None-Match：
- 服务端返回文件内容的hash（ETag）
- 下次请求带上If-None-Match
- 服务端对比hash，相同返回304

【优先级】
ETag > Last-Modified
因为ETag是内容hash，更精准

【🌰完整缓存策略】
// 首次请求
GET /static/app.js HTTP/1.1

// 服务端响应
HTTP/1.1 200 OK
Cache-Control: max-age=31536000
ETag: "abc123"
Last-Modified: Mon, 01 Jan 2024 00:00:00 GMT

// 浏览器缓存这个文件

// 第二次请求（1小时后）
GET /static/app.js HTTP/1.1
If-None-Match: "abc123"
If-Modified-Since: Mon, 01 Jan 2024 00:00:00 GMT

// 服务端发现没变化
HTTP/1.1 304 Not Modified
// 浏览器使用本地缓存

【实际项目配置】
// nginx
location ~* \.(js|css|png|jpg)$ {
  expires 30d;
  add_header Cache-Control "public, no-transform";
}
```

## 5.4 HTTP/HTTPS

### 面试话术

```
【HTTP/1.0 vs 1.1 vs 2.0 vs 3.0】

HTTP/1.1（现在很多项目还在用）：
- 持久连接：TCP连接复用（Connection: keep-alive）
- 管道化：可以发送多个请求，但响应必须按顺序
- 缓存：新增Cache-Control等
- 缺点：队头阻塞，同一TCP连接只能串行请求

HTTP/2.0：
- 二进制分帧：不再是纯文本
- 多路复用：一个TCP连接可以并行多个请求（解决队头阻塞）
- Header压缩：HPACK算法压缩header
- Server Push：服务端主动推送资源
- 缺点：TCP队头阻塞（丢包时所有请求都等）

HTTP/3.0（QUIC协议）：
- 基于UDP：解决TCP队头阻塞
- 0-RTT：首次连接更快
- 连接迁移：切换网络自动恢复
- 缺点：还在普及中

【HTTPS】
HTTPS = HTTP + TLS/SSL

TLS握手过程：
1. 客户端问候：支持的加密算法、随机数A
2. 服务端问候：证书、公钥、随机数B
3. 客户端验证证书，用公钥加密随机数C
4. 服务端用私钥解密C
5. 双方用A+B+C生成对称密钥

后续通信用对称加密（更快）

【为什么要用HTTPS】
1. 数据加密，防窃听
2. 身份认证，防伪装
3. 数据完整性，防篡改
4. SEO加分（Google优先收录）
5. HTTP/2必须HTTPS

【🌰前端常见问题】
1. HTTPS下跨域请求：
   需要CORS配置
   fetch('/api', { credentials: 'include' }) // 携带cookie

2. HTTPS下localhost用HTTP：
   Chrome 94开始，secure contexts默认要求HTTPS
   解决：本地用mkcert生成证书
```

## 6.3 Electron 万级数据虚拟列表

### 面试话术

```
【需求背景】
Electron桌面应用，表格展示万级数据，一次性渲染卡顿严重。

【解决方案：虚拟列表】

虚拟列表核心思想：
只渲染可见区域的数据 + 上下缓冲区，其他区域用空白撑起高度。

【实现原理】
1. 计算可见区域能放多少行：visibleCount = containerHeight / rowHeight
2. 计算滚动位置对应的起始索引：startIndex = Math.floor(scrollTop / rowHeight)
3. 只渲染 startIndex - buffer ~ startIndex + visibleCount + buffer 的数据
4. 用padding撑起总高度：totalHeight = totalCount * rowHeight
5. 滚动时更新startIndex，触发重新渲染

【🌰核心代码】
class VirtualList extends React.Component {
  state = { scrollTop: 0 }

  getVisibleRange() {
    const { scrollTop } = this.state
    const { rowHeight, height, buffer } = this.props
    const visibleCount = Math.ceil(height / rowHeight)
    const startIndex = Math.floor(scrollTop / rowHeight)

    return {
      startIndex: Math.max(0, startIndex - buffer),
      endIndex: startIndex + visibleCount + buffer
    }
  }

  handleScroll = (e) => {
    this.setState({ scrollTop: e.target.scrollTop })
  }

  render() {
    const { scrollTop } = this.state
    const { totalCount, rowHeight, height, renderRow } = this.props
    const { startIndex, endIndex } = this.getVisibleRange()

    const totalHeight = totalCount * rowHeight
    const offsetY = startIndex * rowHeight

    const visibleData = data.slice(startIndex, endIndex)

    return (
      <div
        style={{ height, overflow: 'auto' }}
        onScroll={this.handleScroll}
      >
        <div style={{ height: totalHeight, position: 'relative' }}>
          <div style={{ transform: `translateY(${offsetY}px)` }}>
            {visibleData.map((item, i) => renderRow(item, startIndex + i))}
          </div>
        </div>
      </div>
    )
  }
}

【优化点】
1. 用transform: translateY代替top定位（GPU加速）
2. 用PureComponent或React.memo减少re-render
3. 滚动时用requestAnimationFrame节流
4. 行高固定比行高动态更简单
5. 大列表考虑用react-window库

【项目效果】
支持10万行数据，滚动流畅，内存占用稳定在50MB
```

## 6.4 Web Worker

### 概念理解

Web Worker 是浏览器的"后台线程"，把耗时代码放到后台执行，不阻塞主线程。

### 面试话术

```
【为什么需要Web Worker】
JavaScript单线程，数据计算量大时页面卡死。
Web Worker让你在主线程之外创建worker线程。

【使用方式】

1. 创建worker：
const worker = new Worker('worker.js')

2. 主线程发消息：
worker.postMessage({ type: 'START', payload: data })

3. worker线程接收：
self.onmessage = (e) => {
  const result = heavyComputation(e.data.payload)
  self.postMessage({ type: 'DONE', payload: result })
}

4. 主线程接收结果：
worker.onmessage = (e) => {
  if (e.data.type === 'DONE') {
    console.log(e.data.payload)
  }
}

5. 关闭worker：
worker.terminate()

【worker.js示例 - 大数据排序】
self.onmessage = function(e) {
  const { data } = e.data
  const sorted = data.sort((a, b) => a - b)
  self.postMessage({ result: sorted })
}

【🌰实际场景】
1. 表格排序：10万行数据在worker里排序
2. 加密解密：AES、RSA在worker里
3. 图片处理：Canvas的getImageData处理
4. 数据解析：JSON.parse大文件

【注意事项】
1. Worker不能访问DOM
2. Worker不能访问window对象
3. Worker之间不能直接通信（通过postMessage）
4. Worker里console.log在控制台看不到

【Comlink简化写法】
import * as Comlink from 'comlink'
const worker = Comlink.wrap(new Worker('worker.js'))
const result = await worker.compute(data)  // 像调用函数一样
```


## 7.2 沙箱隔离

### 面试话术

```
【为什么需要沙箱】
多个子应用运行在同一页面，需要隔离：
- JS执行不互相影响
- CSS样式不互相污染

【qiankun沙箱方案】

1. 快照沙箱（legacy）
- 记录window修改前状态
- 子应用运行时记录修改
- 卸载时恢复window

2. 代理沙箱（proxy）
- 创建Proxy代理window
- 子应用对window的操作都在Proxy里
- 多个子应用各自有独立的Proxy

原理：
const fakeWindow = new Proxy(window, {
  get(target, prop) { return storage[prop] ?? target[prop] },
  set(target, prop, value) { storage[prop] = value; return true }
})

【MicroApp沙箱】

1. JS沙箱
- 子应用代码在Proxy环境执行
- 全局操作被拦截
- 不会影响全局window

2. CSS沙箱
- Shadow DOM：天然样式隔离
- CSS变量前缀：统一添加前缀

【🌰样式隔离示例】
基座定义：
:host {
  --app-primary-color: blue;
}

子应用：
.element {
  color: var(--app-primary-color);  // 正常访问
  --micro-color: red;  // 被改名为--micro-color
}

【注意事项】
1. 快照沙箱不支持多实例（多个子应用同时运行）
2. 代理沙箱需要额外内存
3. 不要在沙箱外引用沙箱内的变量
```

## 7.4 低代码表单：JSON Schema 设计

### 面试话术

```
【需求背景】
可视化表单编辑器，用户拖拽配置，生成JSON Schema，后端存储JSON，前端渲染。

【JSON Schema标准】
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "properties": {
    "name": {
      "type": "string",
      "title": "姓名",
      "rules": [{ "required": true }, { "pattern": "^[\u4e00-\u9fa5]{2,4}$", "message": "请输入2-4位中文" }]
    },
    "email": {
      "type": "string",
      "title": "邮箱",
      "rules": [{ "required": true }, { "type": "email", "message": "邮箱格式不正确" }]
    },
    "age": {
      "type": "number",
      "title": "年龄",
      "rules": [{ "required": true }, { "min": 0 }, { "max": 150 }]
    }
  },
  "required": ["name", "email"]
}

【设计要点】

1. 组件映射
每种字段类型对应一个组件：
{
  type: 'input',
  props: { placeholder: '请输入' }
}

{
  type: 'select',

      { label: '选项1', value: 1 },

    ]
  }

{
  type: 'select',
  name: 'province',
  dataSource: '/api/provinces'
},
{
  type: 'select',
  name: 'city',
  // 联动：依赖province变化

  dataSource: '/api/cities?province={{province}}'
}

3. 动态字段
{
  type: 'array',
  name: 'contacts',
  itemSchema: {
    type: 'object',
    properties: {
      name: { type: 'input' },
      phone: { type: 'input' }
    }
  }
}

【渲染实现】
使用 ajv 做JSON Schema校验
使用 Vue/React 根据Schema动态渲染组件

【🌰简化代码】
const schemas = {
  input: (props) => <Input {...props} />,
  select: (props) => <Select {...props} />
}

function FormRender({ schema }) {
  const Component = schemas[schema.type]
  return (
    <Form.Item label={schema.title}>
      <Component {...schema.props} />
    </Form.Item>
  )
}
```

### 2. 实现一个防抖函数

```javascript
function debounce(fn, delay, immediate = false) {
  let timer = null;

  return function (...args) {
    const context = this;

    if (timer) clearTimeout(timer);

    if (immediate && !timer) {
      fn.apply(context, args);
    }

    timer = setTimeout(() => {
      if (!immediate) {
        fn.apply(context, args);
      }
      timer = null;
    }, delay);
  };
}

// 使用
const handleSearch = debounce((value) => {
  console.log("搜索:", value);
}, 300);

input.addEventListener("input", (e) => {
  handleSearch(e.target.value);
});
```

### 3. 实现深拷贝

```javascript
function deepClone(obj, map = new WeakMap()) {
  // 基本类型
  if (obj === null || typeof obj !== "object") {
    return obj;
  }

  // 循环引用
  if (map.has(obj)) {
    return map.get(obj);
  }

  // Date
  if (obj instanceof Date) {
    return new Date(obj);
  }

  // RegExp
  if (obj instanceof RegExp) {
    return new RegExp(obj.source, obj.flags);
  }

  // Map
  if (obj instanceof Map) {
    const clone = new Map();
    map.set(obj, clone);
    obj.forEach((v, k) => {
      clone.set(deepClone(k, map), deepClone(v, map));
    });
    return clone;
  }

  // Set
  if (obj instanceof Set) {
    const clone = new Set();
    map.set(obj, clone);
    obj.forEach((v) => {
      clone.add(deepClone(v, map));
    });
    return clone;
  }

  // Array / Object
  const clone = Array.isArray(obj) ? [] : {};
  map.set(obj, clone);

  for (const key in obj) {
    if (obj.hasOwnProperty(key)) {
      clone[key] = deepClone(obj[key], map);
    }
  }

  return clone;
}
```

### 4. 实现 Promise.all

```javascript
function promiseAll(promises) {
  return new Promise((resolve, reject) => {
    const results = [];
    let completedCount = 0;

    if (promises.length === 0) {
      resolve([]);
      return;
    }

    promises.forEach((promise, index) => {
      Promise.resolve(promise).then(
        (value) => {
          results[index] = value;
          completedCount++;
          if (completedCount === promises.length) {
            resolve(results);
          }
        },
        (error) => {
          reject(error);
        }
      );
    });
  });
}
```

## React Hooks

```
useState存状态，每次渲染链表找
useEffect副作用，cleanup要记牢
useCallback记函数，useMemo记结果
useRef盒子不变，盒子内容随便改
```

## Event Loop

```
同步代码先执行
微任务接着执行
DOM渲染后执行
宏任务最后执行
```

## 作用域链

```
变量查找往外走，走到全局都没有
闭包就是记住它，出生时的外部变量
```

```
防抖等停手，只执行最后
节流控频率，固定间隔走
```
