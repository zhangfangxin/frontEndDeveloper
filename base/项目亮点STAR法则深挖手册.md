### 🔥 面试官深挖问题

#### 技术细节类

**Q1：MicroApp的原理是什么？沙箱是怎么实现的？**

标准答案：
> MicroApp的原理是基于WebComponent的customElement特性创建隔离的DOM容器，通过拦截window操作实现JS沙箱。
> 
> 具体来说，沙箱实现分为三部分：
> 1. **JS沙箱**：通过Proxy拦截window.addEventListener、window.removeEventListener等全局操作，每个子应用有独立的快照和变更记录
> 2. **CSS沙箱**：在子应用容器上添加唯一的data-element-root属性，CSS选择器自动带上这个前缀实现样式隔离
> 3. **通信机制**：基于发布订阅模式，子应用通过window.dispatchEvent发送消息，主应用通过customEvent接收
> 
> 我当时对比了qiankun，qiankun基于single-spa，采用快照沙箱，原理是对原生window进行深拷贝保存快照，子应用运行后再恢复。MicroApp的沙箱更轻量，是通过Proxy拦截增量操作，不需要对window整体拷贝。

**Q2：DllPlugin是如何减少构建时间的？**

标准答案：
> DllPlugin是Webpack内置的预编译方案，原理是将不会频繁改动的第三方库抽离成独立的bundle。
> 
> 具体流程：
> 1. 新建webpack.dll.config.js，配置需要预编译的库（React、ReactDOM、AntDesign等）
> 2. 使用DllPlugin生成manifest.json映射文件
> 3. 主配置使用DllReferencePlugin引用映射文件
> 4. 构建时直接使用缓存的dll文件，不再重复解析
> 
> 效果：首次构建会多花30秒生成dll，但之后的每次构建，这些库不需要再被解析和打包，所以构建时间直接砍半。

**Q3：JSON Schema驱动的表单引擎是怎么设计的？**

标准答案：
> 我的设计思路是"配置即代码"，核心是schema解析器和渲染引擎。
> 
> schema结构分为三层：
> 1. **基础层**：定义字段类型（input、select、datepicker等）、校验规则（required、pattern、minLength等）
> 2. **联动层**：定义字段间依赖关系，比如"选择A后B字段显示且必填"
> 3. **布局层**：定义栅格、折叠面板、分组等布局结构
> 
> 渲染引擎的核心逻辑是：解析schema生成虚拟DOM树，通过React的render函数逐层渲染。联动逻辑通过观察者模式实现，字段值变化时通知所有依赖它的字段重新渲染。
> 
> 关键是实现了模板市场，沉淀了50+业务模板，新表单开发只需要组合模板+调整少量字段。

**Q4：pnpm的monorepo是怎么配置的？**

标准答案：
> pnpm的workspace机制是核心。我们项目结构是：
> ```
> /packages
>   /common-components  # 公共组件
>   /utils              # 工具函数
>   /types              # TS类型定义
> /apps
>   /main-app           # 主应用
>   /child-app-1        # 子应用1
>   /child-app-2        # 子应用2
> ```
> 
> pnpm-workspace.yaml配置：`packages: ['packages/*', 'apps/*']`
> 
> 优势：
> - 所有子项目的公共依赖提升到根目录，只安装一次
> - 通过软链接复用内部包，修改后实时生效
> - 避免幽灵依赖，依赖关系透明清晰

#### 挑战困难类

**Q5：微前端改造过程中，遇到最大的技术挑战是什么？**

标准答案：
> 最大的挑战是**样式冲突问题**。
> 
> 当时有3个子应用都用了AntDesign，有的版本是3.x，有的是4.x，样式互相污染。我试过几种方案：
> 
> 方案1：CSS Modules——需要改所有组件的className，工作量太大，放弃
> 方案2：shadow DOM——对第三方组件样式支持差，AntDesign的动态样式不生效
> 方案3：CSS变量隔离——给每个子应用注入不同的CSS变量前缀，但AntDesign不友好
> 
> 最终方案：统一所有子应用到AntDesign 4.x版本，然后通过MicroApp的CSS沙箱机制，在运行时给所有选择器注入子应用专属的data属性选择器前缀。这样不用改源码，实现了零侵入式隔离。

**Q6：构建优化过程中，有没有什么坑？**

标准答案：
> 有个印象深刻的坑是**缓存失效问题**。
> 
> 开启webpack5持久化缓存后，第一次构建正常，但修改代码后重新构建，很多文件没有更新，页面显示旧代码。
> 
> 排查了很久，发现原因是：
> 1. 我们的node_modules是通过pnpm的软链接指向全局store
> 2. webpack缓存是基于文件路径计算的，但软链接导致不同子应用的相同依赖文件路径不一致
> 3. 缓存的hash值计算出错
> 
> 解决方案：配置webpack的cache.managedPaths，将pnpm的软链接路径纳入缓存管理；同时给每个子应用设置独立的cache目录。

**Q7：表单引擎在实际使用中遇到过什么bug？**

标准答案：
> 遇到过**联动逻辑的循环依赖问题**。
> 
> 场景是：字段A依赖B，B又依赖A。比如"选择省之后显示市，选择市之后显示区"，但如果配置不当，可能出现"选了省，Areas被清空，导致City也被清空"的连锁反应。
> 
> 解决思路：
> 1. 设计了一套联动优先级规则：按字段在schema中的声明顺序
> 2. 增加依赖图的检测算法，启动时检测是否存在环
> 3. 采用批处理更新模式：收集所有变更后统一渲染，避免中间状态的渲染

#### 决策思考类

**Q8：为什么选MicroApp而不是qiankun？**

标准答案：
> 我做了详细的技术对比：
> 
> | 维度 | qiankun | MicroApp |
> |------|---------|----------|
> | 原理 | single-spa + 快照沙箱 | WebComponent + 拦截沙箱 |
> | Vue3支持 | 需要手动配置 | 天然支持 |
> | CSS隔离 | 需要手动配置CSS Modules | 自动注入前缀 |
> | API复杂度 | 较高，需要理解生命周期 | 简洁，类iframe写法 |
> | 团队成本 | 学习曲线陡峭 | 上手快 |
> 
> 我们团队当时Vue3项目占比60%，MicroApp对Vue3的天然支持和低学习成本是决定因素。另外qiankun的快照沙箱在大量子应用场景下会有性能问题，而MicroApp的沙箱是增量式的，更适合我们的规模。

**Q9：为什么从yarn切换到pnpm？**

标准答案：
> 核心原因是**磁盘空间和安装速度**。
> 
> 当时我们的node_modules有1.2GB，每周CI构建都要重新下载一次，磁盘IO成为瓶颈。yarn的node_modules是扁平化的，存在幽灵依赖问题——A依赖B v1，但实际跑的是B v2，因为C也依赖B v2，npm会提升C的版本。
> 
> pnpm的硬链接机制：同一个依赖全局只存一份，通过硬链接给各项目使用，磁盘空间节省60%。安装速度也快了40%。
> 
> 另外pnpm的严格依赖管理帮我排查出了一个版本冲突bug：项目里同时存在两个版本的lodash，虽然都能用，但会导致某些工具函数行为不一致。

**Q10：低代码表单 vs 使用formily，你是怎么选的？**

标准答案：
> 我调研了formily、react-jsonschema-form、低代码自研三条路。
> 
> formily功能强大但过于重，学习成本高，而且扩展定制要深入源码，改动风险大。
> react-jsonschema-form符合JSON Schema标准，但样式定制困难，不符合业务需要的金融系统风格。
> 
> 最终选择自研，因为：
> 1. 我们的业务场景特殊：金融表单有复杂的联动校验、金额格式化、日期范围联动等
> 2. 自研可以完全掌控，可以随时调整
> 3. 核心逻辑其实不复杂，2周就能出MVP
> 
> 原则是：80%场景用通用方案，20%场景特殊处理。自研的表单引擎解决了20%的特殊场景，通用场景用组合配置实现。

### 🔥 面试官深挖问题

#### 技术细节类

**Q1：Electron的主进程和渲染进程是怎么通信的？**

标准答案：
> Electron有三种IPC通信方式：
> 
> **1. ipcMain/ipcRenderer（双向通信）**
> 渲染进程发消息：`ipcRenderer.send('channel', data)`
> 主进程监听：`ipcMain.on('channel', (event, data) => {...})`
> 主进程回复：`event.reply('response', result)`
> 
> **2. invoke/handle（Promise风格，推荐）**
> 渲染进程：`const result = await ipcRenderer.invoke('get-path', 'desktop')`
> 主进程：`ipcMain.handle('get-path', async (event, name) => { return app.getPath(name) })`
> 
> **3. contextBridge（安全暴露API）**
> 在preload脚本里暴露安全的API给渲染进程，避免直接暴露node模块
> 
> 我们项目的最佳实践是：普通单向通知用send/on，复杂数据交互用invoke/handle，通过contextBridge封装所有主进程能力，渲染进程不能直接访问Node.js API。

**Q2：虚拟列表是怎么实现的？**

标准答案：
> 虚拟列表的核心原理是"只渲染可视区域内的元素"。
> 
> 具体实现思路：
> 1. 计算可视区域能容纳的元素数量：可视高度 / 单项高度
> 2. 计算需要渲染的起始索引：scrollTop / 单项高度
> 3. 在列表容器上方放一个同等总高度的占位div撑开滚动条
> 4. 只渲染起始索引到起始索引+可视数量的元素
> 5. 滚动时动态计算并更新渲染范围
> 
> react-window的用法：
> ```jsx
> <FixedSizeList
>   height={600}
>   itemCount={10000}
>   itemSize={50}
>   width="100%"
> >
>   {({ index, style }) => (
>     <div style={style}>
>       {/* 只渲染当前index的元素 */}     </div>
>   )}
> </FixedSizeList>
> ```
> 
> 我们对react-window做了二次封装，增加了动态行高、无限滚动加载、选中高亮等功能。

**Q3：Web Worker是怎么使用的？**

标准答案：
> Web Worker让JS在后台线程运行，不阻塞主线程。
> 
> 使用流程：
> 1. **创建Worker文件**：`utils.worker.js`，在里面写计算逻辑
> ```javascript
> self.onmessage = ({ data }) => {
>   const result = heavyCalculation(data);
>   self.postMessage(result);
> }
> ```
> 
> 2. **在主线程调用**：
> ```javascript
> const worker = new Worker(new URL('./utils.worker.js', import.meta.url));
> worker.postMessage({ list: bigArray, filter: 'xxx' });
> worker.onmessage = ({ data }) => {
>   // data是计算结果，直接更新状态
>   setFilteredList(data.result);
> };
> ```
> 
> 我们把数据聚合（求和、分组、排序）、大数据量筛选、复杂计算都放Worker里。5000条数据聚合计算，主线程需要80ms，Worker只需要5ms，用户操作完全感觉不到延迟。

**Q4：TypeScript strict模式的strict都包含什么？**

标准答案：
> TypeScript的strict模式包含多个检查选项：
> 
> - **strictNullChecks**：严格检查null和undefined，不能直接赋值给其他类型
> - **noImplicitAny**：不允许推断为any的类型，必须显式声明
> - **strictFunctionTypes**：函数参数类型必须严格匹配
> - **strictBindCallApply**：bind/call/apply必须类型正确
> - **strictPropertyInitialization**：类的属性必须在构造函数中初始化
> - **noImplicitThis**：this必须类型明确
> 
> 开启strict后，很多之前被忽略的小问题都会报出来，比如：
> - `const fn = (arr) => arr.filter(x => x > 0)` 会报错，因为x可能是undefined
> - `document.getElementById('xxx').value` 会报错，因为可能是null
> 
> 迁移成本确实有，但长期收益很大。我们花了2周时间处理所有报错，换来的是重构安全性大幅提升——改一个公共类型，全项目类型不兼容的地方都会报错，而不是运行时才暴露。

**Q5：Electron打包后遇到过什么兼容性问题？**

标准答案：

**Q6：大数据量下的性能瓶颈是怎么定位的？**

> - 大数据量场景用Web Worker计算

> 2. **Node生态**：项目依赖了大量npm包（ECharts、AntDesign、lodash等），换成Tauri需要重写
> 3. **开发效率**：Electron的开发体验更成熟，调试工具更完善
> 4. **当时Tauri还不够成熟**：2022年Tauri刚出1.0，生态不如Electron完善
>

**Q9：Electron应用还有什么可以优化的点？**

标准答案：

> 4. **多窗口优化**：目前多个窗口各自独立渲染，可以考虑共享渲染进程，减少内存占用

> 5. **安全加固**：目前用sessionStorage，可以升级到electron-store的加密存储；IPC通信可以增加签名验证

### 🔥 面试官深挖问题

#### 技术细节类

**Q1：Sentry是怎么实现源码定位的？**

标准答案：
> 关键是SourceMap的生成和上传。
> 
> 流程：
> 1. 开发时：Webpack构建生成`.js.map`文件（SourceMap）
> 2. 构建时：上传SourceMap到Sentry服务器（或者用sentry-cli在CI/CD中自动上传）
> 3. 用户访问：代码压缩后部署，JS里的变量名、源码行列信息都被删除
> 4. 报错时：浏览器上报的是压缩后的错误位置（只有行号，没有列号）
> 5. Sentry服务端：用上传的SourceMap反向映射，把压缩后的位置映射回原始源码位置
> 
> 示例：
> 压缩前：`const userName = response.data.user.name`
> 压缩后：`const n=r.data.u.n`
> 报错时：Sentry根据SourceMap还原出原始代码和行列号
> 
> 安全注意：生产环境SourceMap要私有化，不能公开暴露，否则用户可以反编译看到源码。

**Q2：用户行为回放是怎么实现的？**

标准答案：
> rrweb（record and replay the web）是业界常用方案。
> 
> 实现原理：
> 1. **录制阶段**：监听DOM变化事件（mutation observer）、鼠标事件、输入事件、滚动事件等，序列化成JSON
> 2. **压缩存储**：事件很多，需要去重、合并、压缩，比如连续多次scroll只记录起止位置
> 3. **回放阶段**：用rrweb-player播放，按时间线重放所有操作
> 
> 数据结构示例：
> ```json
> [
>   { "type": 4, "data": { "source": 2, "position": { "x": 100, "y": 200 } } },
>   { "type": 3, "data": { "source": 1, "text": "关键词" } }
> ]
> ```
> 
> 我们用它定位了一个困扰了2周的bug：用户说"点了按钮没反应"，回放发现是按钮点击后网络请求超时，但UI没有显示错误提示。纯靠错误日志根本发现不了。

**Q3：Elasticsearch全文检索是怎么接入的？**

标准答案：
> 前端侧的实现：
> 
> 1. **数据同步**：后端定期同步数据到ES，我们只负责调用搜索接口
> 2. **搜索接口**：封装统一搜索服务，支持关键词高亮、分页、过滤
> ```javascript
> const result = await searchAPI({
>   query: '如何开通账户',
>   index: 'knowledge_base',
>   filters: { category: 'account', status: 'published' },
>   highlight: true,
>   from: 0,
>   size: 20
> });
> ```
> 3. **高亮渲染**：ES返回的`<em>`高亮标签，前端用v-html渲染
> 
> 分词策略：中文用IK分词器，支持词根匹配。比如搜"开户"能匹配到"开立账户"。

**Q4：jQuery迁移到Vue是怎么做的？**

标准答案：
> 迁移策略是"渐进式替换"：
> 
> 1. **共存阶段**：不删除jQuery代码，让Vue和jQuery同时运行
> 2. **逐模块替换**：按功能模块替换，一个模块一个模块来
> 3. **建立桥接层**：公共方法封装成Vue插件或mixin
> 
> 替换原则：
> - DOM操作频繁的 → Vue组件
> - 纯展示的静态页面 → Vue模板
> - 工具函数 → 独立JS模块
> - 第三方jQuery插件 → 封装成Vue组件（用ref获取DOM）
> 
> 关键经验：
> - 先建立状态管理（Vuex），把共享数据收拢
> - 编写《从jQuery到Vue的迁移手册》，统一编码规范
> - 设置Eslint规则，禁止新的jQuery代码写入

> 关键成果是支持韩文、中文、英文三种语言自由切换，用户数从零起步逐步增长到日活跃5万。同时优化了WebSocket协议使带宽占用降低60%，并实现了交易页面的毫秒级响应。

### 🔥 面试官深挖问题

#### 技术细节类

**Q1：WebSocket为什么需要心跳机制？**

标准答案：
> 心跳机制有两个作用：
> 
> 1. **检测连接存活**：网络不是永恒的，NAT超时、路由器重启、中间节点断连都会导致连接"假死"——服务端和客户端都认为连接正常，但实际上已经不通了。
> 
> 2. **保持连接活跃**：有些防火墙、负载均衡会断开长时间空闲的连接。心跳可以保持连接活跃。
> 
> 实现方案：
> ```javascript
> // 客户端：每30秒发一次心跳
> setInterval(() => {
>   if (ws.readyState === WebSocket.OPEN) {
>     ws.send(JSON.stringify({ type: 'ping' }));
>   }
> }, 30000);
> 
> // 服务端：60秒没收到心跳则断开
> // 客户端：收不到pong则重连
> ```
> 
> 重连策略用指数退避：首次1秒，再失败2秒、4秒、8秒...最多30秒，避免频繁重连。

**Q2：WebSocket多路复用是怎么实现的？**

标准答案：
> 多路复用的核心是：一个TCP连接上传输多个数据通道。
> 
> 实现思路：
> 1. **协议设计**：每条消息有个channel字段标识频道
> ```javascript
> { channel: 'btc_usdt', type: 'ticker', data: { price: 50000 } }
> { channel: 'eth_usdt', type: 'ticker', data: { price: 3000 } }
> ```
> 
> 2. **订阅管理**：客户端维护一个订阅列表
> ```javascript
> const subscriptions = new Set(['btc_usdt', 'eth_usdt']);
> 
> function subscribe(symbol) {
>   subscriptions.add(symbol);
>   ws.send(JSON.stringify({ action: 'subscribe', symbols: [symbol] }));
> }
> ```
> 
> 3. **服务端推送**：一个WebSocket连接推送所有频道数据，客户端按channel过滤
> 
> 优势：比建多个WebSocket连接省资源，比轮询HTTP实时性高，比普通WebSocket管理简单。

**Q3：TradingView图表是怎么集成的？**

标准答案：
> 我们用的是TradingView的lightweightCharts库。
> 
> 集成步骤：
> 1. 安装：`npm install lightweight-charts`
> 2. 创建图表容器
> 3. 添加K线series：`chart.addCandlestickSeries()`
> 4. 绑定WebSocket数据
> ```javascript
> const chart = createChart(container, { width: 800, height: 400 });
> const candleSeries = chart.addCandlestickSeries({
>   upColor: '#26a69a',
>   downColor: '#ef5350'
> });
> 
> ws.onmessage = (e) => {
>   const data = JSON.parse(e.data);
>   candleSeries.update({
>     time: data.timestamp,
>     open: data.open,
>     high: data.high,
>     low: data.low,
>     close: data.close
>   });
> };
> ```
> 
> 技术指标用内置的MACD、RSI插件，性能优化是把历史K线分段加载，滚动到边界再加载下一段。

**Q4：数据防抖节流是怎么实现的？**

标准答案：
> 场景是：WebSocket推送可能是100ms一条，但渲染跟不上这个频率。
> 
> **防抖（Debounce）**：等用户停止输入后才执行
> ```javascript
> function debounce(fn, delay) {
>   let timer = null;
>   return (...args) => {
>     clearTimeout(timer);
>     timer = setTimeout(() => fn(...args), delay);
>   };
> }
> ```
> 
> **节流（Throttle）**：固定时间执行一次
> ```javascript
> function throttle(fn, limit) {
>   let inThrottle = false;
>   return (...args) => {
>     if (!inThrottle) {
>       fn(...args);
>       inThrottle = true;
>       setTimeout(() => inThrottle = false, limit);
>     }
>   };
> }
> ```
> 
> 我们的实践：
> - K线数据：节流200ms，限制渲染频率
> - 深度图：节流500ms，深度图不需要那么频繁
> - 最新成交：防抖100ms，避免闪屏

#### 挑战困难类

**Q5：WebSocket断线重连是怎么处理的？**

标准答案：
> 断线重连需要考虑三个问题：检测、策略、恢复。
> 
> **1. 断线检测**
> - `onclose`、`onerror`事件直接触发重连
> - 心跳超时检测：收不到pong就认为断了
> 
> **2. 重连策略**
> ```javascript
> function connect() {
>   ws = new WebSocket(url);
>   ws.onclose = () => {
>     // 指数退避：1s -> 2s -> 4s -> 8s...上限30s
>     const delay = Math.min(30000, baseDelay * Math.pow(2, attempt));
>     attempt++;
>     setTimeout(connect, delay);
>   };
> }
> ```
> 
> **3. 状态恢复**
> - 记录订阅列表，断线重连后重新订阅
> - 缓存最近数据，重连期间不显示空白
> - 重连成功通知用户"正在重新连接..."
> 
> 特殊处理：页面后台时暂停WebSocket（节省资源），切回来再重连。

**Q6：K线图加载10万条数据怎么优化？**

标准答案：
> 10万条K线是很大的数据量，全部渲染会卡死。
> 
> 优化方案：
> 1. **分片加载**：只加载可视区域及其附近的K线
> ```javascript
> // 滚动时动态加载
> chart.timeScale().subscribeVisibleLogicalRangeChange(range => {
>   if (range.to - range.from > threshold) {
>     loadMoreData(direction);
>   }
> });
> ```
> 
> 2. **数据降采样**：时间范围大时，用1小时K线替代1分钟K线
> ```javascript
> function downsample(data, factor) {
>   return data.filter((_, i) => i % factor === 0);
> }
> ```
> 
> 3. **懒加载优化**：用IntersectionObserver，只在图表进入可视区才渲染
> 
> 4. **WebWorker计算**：聚合计算在Worker里做，不阻塞主线程
> 
> 最终效果：10万条K线滚动流畅，首屏加载时间控制在1秒内。

#### 决策思考类

**Q7：为什么选TradingView而不是自己画K线？**

标准答案：
> 选TradingView有三个原因：
> 
> 1. **专业性**：TradingView是业界标准，K线渲染、交互体验都是顶级水准。自己画很难达到这个水准。
> 
> 2. **开发效率**：交易所功能复杂，需要把精力放在业务逻辑上。TradingView有现成的K线、深度图、技术指标，可以快速集成。
> 
> 3. **维护成本**：K线图是个无底洞，各种边界情况（缩放、拖动、指标叠加）都要处理。用成熟的库，省去大量维护工作。
> 
> 缺点是要付费（专业版），但换来的是开发效率和产品质量，这个投入是值得的。

**Q8：vue-i18n是怎么组织的？**

标准答案：
> 目录结构：
> ```
> /locales
>   /ko.json      # 韩文
>   /zh.json      # 中文
>   /en.json      # 英文
> /views
>   /trade
>     /order.json  # 交易模块翻译
>   /user
>     /profile.json # 用户模块翻译
> ```
> 
> key命名：`模块_页面_功能_状态`
> 示例：`trade.order.form.submit.success`
> 
> 复用策略：把通用的翻译抽到公共文件
> ```json
> {
>   "common": {
>     "confirm": "确认",
>     "cancel": "取消",
>     "loading": "加载中..."
>   }
> }
> ```
> 
> 切换实现：
> ```javascript
> i18n.locale = localStorage.getItem('locale') || 'ko';
> ```
> 
> 经验：翻译文件和业务代码要分离，不要在组件里写死文案。

## 📌 面试通用技巧

### 1. STAR法则使用技巧
- **S和T要简洁**：背景和职责说清楚就行，不要超过1分钟
- **A要突出技术决策**：面试官最想知道你"怎么想的"
- **R要用数据说话**：数字比文字更有说服力

### 2. 深挖问题应对策略
- **技术细节**：提前准备原理层面的理解，不能只说"用了这个工具"
- **挑战类**：准备1-2个印象深刻的问题，说明解决思路
- **决策类**：说明你考虑了哪些方案，最终选这个的原因
- **优化类**：承认有改进空间，但说明理由

### 3. 话术长度控制
- **30秒版本**：只说亮点（技术栈+量化结果）
- **1分钟版本**：STAR各说一句话
- **3分钟版本**：每个项目完整STAR + 1-2个深挖问题的详细回答

### 4. 注意事项
- 不要编造数据，项目里没做过的功能不要硬扯
- 不确定的细节不要说死，用"大概"、"可能"等词
- 保持自信，即使问到不会的也可以说"这个我没深入了解，但我的理解是..."
